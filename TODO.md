# simplesolat-data — Implementation Plan

## Overview

Centralized prayer times data repository. CI fetches from upstream sources and commits JSON files. The simplesolat-api sync reads from this repo instead of hitting upstream APIs directly. Community can contribute new countries via PRs.

## Directory Structure

```

  data/                            # all consumable data for downstream (API, mobile)
    countries.yaml                 # official country definitions with geojson/mapping URLs

      malaysia-adm2-{date}.json
      singapore-adm0-{date}.json

      sri-lanka-adm2-{date}.json
    mappings/                      # derived — generated by scripts/generate_mappings.py
```

## Data Format

Each file is one zone, one month. Array of daily records:

```json
[
  {
    "date": "2026-01-01",
    "imsak": "06:01",
    "fajr": "06:11",
    "syuruk": "07:17",
    "dhuhr": "13:24",
    "asr": "16:30",
    "maghrib": "19:26",
    "isha": "20:35"
  },
  ...
]
```

- Times are HH:MM in local timezone (no seconds, no timezone info)
- Timezone is per-zone in data/zones/{CC}.yaml (IANA timezone, e.g. Asia/Kuala_Lumpur)
- API converts local HH:MM → epoch using zone timezone when serving to clients
- 28–31 records per file (one per day of the month)

## Zone File Format

Per-country zone files (`data/zones/{CC}.yaml`):

```yaml
zones:
  - code: JHR02
    country: MY
    state: Johor
    location: Johor Bahru, Kota Tinggi, Mersing, Kulai
    timezone: Asia/Kuala_Lumpur
    shapes:
      - Johor Bahru
      - Kota Tinggi
      - Kulai
      - Mersing
```

- `code` — zone code (JAKIM codes for MY, convention for others)
- `country` — ISO country code
- `state` — administrative region (display, grouping)
- `location` — upstream description of area covered (display)
- `timezone` — IANA timezone for time conversion
- `shapes` — geojson shapeNames this zone covers (used to derive mapping files)

Mapping files (`data/mappings/*.json`) are derived artifacts — generated by `scripts/generate_mappings.py` from zone and geojson data.

- Check what data already exists in data/prayer-times/

- Be idempotent (safe to re-run)
- Handle errors gracefully (log and continue)

Source-specific quirks (KHEU minute offsets, ACJU no-imsak derivation, KHEU dot-separated times) are handled within each script, not in zone definitions.

## Upstream API Notes

### JAKIM (Malaysia)
- POST https://www.e-solat.gov.my/index.php?r=esolatApi/takwimsolat
- Per zone, per year date range
- 1s throttle between requests
- 60 zones × 2 years = ~120 requests

### MUIS (Singapore)
- GET https://data.gov.sg/api/action/datastore_search
- Single bulk fetch, ~1100 records
- Needs API key (MUIS_API_KEY) for higher rate limits
- 1 request total

### EQuran (Indonesia)
- POST https://equran.id/api/v2/shalat
- Per zone per month, parallel (10 workers, no rate limiting observed)
- 517 zones × 12 months = ~6200 requests per year (~7 min with parallel)
- Returns 404 for unavailable data (3 zones have no data)
- Request body: {"provinsi": zone.state, "kabkota": zone.location, "bulan": month, "tahun": year}

### KHEU (Brunei)
- Using Taqwim PDF from https://www.mora.gov.bn (full year, published annually)
- PDF URL in sources/kheu/sources.yaml, downloaded on demand
- 4 zones with minute offsets: BRN01 (base), BRN02 (+1min), BRN03 (+3min), BRN04 (base)
- Considered SharePoint REST API (mora.gov.bn/_api/web/lists/...) but decided against it:
  flaky DNS, only publishes current/next month, dot-separated 12-hour times without AM/PM,
  upstream typos ("112.28", "741"). The PDF is the authoritative source, has full year data,
  and avoids all these issues.

### ACJU (Sri Lanka)
- No API — PDFs published yearly at https://www.acju.lk/prayer-times/
- 13 official zones (multiple districts share same times)
- PDF URLs committed in sources/acju/sources.yaml, downloaded on demand
- Extracted via scripts/fetch_acju.py (pdfplumber, parallel download)
- No imsak — derived as fajr - 10 min

## Caching Strategy

| Data | Mobile cache | Invalidation |
|------|-------------|-------------|
| Countries list | 1 month | Refetch from API monthly |
| GeoJSON / mappings | Indefinite | URL change (datestamp) triggers delete old + fetch new |
| Zones per country | 1 month | Refetch from API monthly |
| Prayer times | Current + next month | Trimmed on each fetch |

## Zone Code Convention

- **MY**: JAKIM's official codes (JHR01, SGR01, WLY01, etc.)
- **SG**: SGP01 (single zone)
- **ID**: Province prefix + number (ACH01, JKT01, etc.) — generated convention
- **BN**: BRN01-04
- **LK**: LK01-13

## Contributing a New Country

1. Find official prayer times source (government authority, PDF, API)
2. Write a script in `scripts/` that outputs JSON in the data format above
3. Add country to `data/countries.yaml` with geojson/mapping URLs
4. Add zone definitions to `data/zones/{CC}.yaml` with shapes
5. Run `scripts/generate_mappings.py` to derive mapping file
6. Add datestamped geojson file to `data/geojson/`
7. If source is PDF, add source URLs to `sources/` and extraction script
8. Submit PR with all of the above
